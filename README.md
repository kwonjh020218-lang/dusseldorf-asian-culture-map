# 🇩🇪🎌 뒤셀도르프 아시안 컬처 맵

뒤셀도르프의 한인·아시안 맛집, 마트, 만화카페, 서점, 명소, 굿즈샵 등 **아시아 문화를 즐길 수 있는 122곳**을 지도에서 찾아보고, 나만의 코스를 짤 수 있는 웹앱입니다.

**🔗 라이브 데모: [바로 열어보기](https://kwonjh020218-lang.github.io/dusseldorf-asian-culture-map/)**

## 왜 만들었나요

뒤셀도르프에는 유럽 최대 규모의 일본인 커뮤니티가 있는 "리틀 도쿄"(임머만 거리)가 있어서, 한식·일식·중식·베트남·태국 음식점부터 만화카페, 일본 서점, K-pop 굿즈샵까지 다양한 아시아 문화 스팟이 밀집해 있습니다. 하지만 이 정보를 한눈에 볼 수 있는 곳이 없어서 직접 지도를 만들었습니다.

바이브 코딩(AI 페어 프로그래밍)으로 기획부터 배포까지 진행하며, 실제 사용자 관점에서 필요한 기능을 계속 다듬어나간 프로젝트입니다.

## 주요 기능

- 🔍 **검색 & 필터**: 이름 검색, 13개 카테고리(국가별 음식 + 마트/카페/오락/서점/디저트/명소/굿즈), 가격대, 비건/매운맛/포토스팟 여부
- 📍 **실시간 위치 기반**: 내 위치에서 거리순 정렬, 근처 카페·디저트 자동 추천
- 🧭 **나만의 코스 빌더**: 팝업에서 코스에 담고 순서 조정, 담을 때마다 자동으로 도보 경로가 그려짐 (내 위치 또는 뒤셀도르프 중앙역 기준)
- 🎲 **추천 코스**: 필터 조건을 존중하며 "식당 → 카페/디저트 → 놀거리"를 자동으로 골라주는 기능
- ⭐ **개인화**: 즐겨찾기, 방문 체크, 방문 수에 따른 레벨 뱃지 (localStorage 저장)
- 🗺️ **지도 UX**: 마커 클러스터링, 호버 툴팁, 클릭 시 마커 강조 애니메이션
- 🎲 **오늘 뭐 하지?**: 현재 필터 기준으로 랜덤 추천해주는 룰렛
- 🌐 **다국어(i18n)**: 한국어/English/Deutsch 지원. UI 텍스트뿐 아니라 122곳의 설명·대표메뉴까지 전체 번역됨 (터키어·아랍어는 추후 추가 예정 - 아랍어는 텍스트 방향이 반대(RTL)라 레이아웃 작업이 추가로 필요함)

## 기술 스택

- **순수 HTML/CSS/JavaScript (프론트엔드)** — 프레임워크·번들러 없이 바닐라 JS로 구현
- **[Supabase](https://supabase.com/) (백엔드/DB)** — PostgreSQL 데이터베이스에 122곳 데이터를 저장하고, REST API로 fetch해서 가져오는 구조. 원래는 코드 안에 데이터가 직접 박혀있었는데, 코드와 데이터를 분리하고 실제 DB 연동 구조로 리팩토링함
- **자체 제작 i18n(다국어) 레이어** — UI 문구·카테고리명은 `i18n.js`에 정적으로 두고, **장소별 설명/대표메뉴 번역은 `place_translations` 테이블에 별도로 저장**해서 `places` 테이블과 외래키(FK)로 연결. Supabase가 관계를 자동으로 인식해서, 장소 데이터를 요청할 때 번역까지 한 번에 묶어서(nested) 내려줌 (`places?select=*,place_translations(*)`)
- **[Leaflet.js](https://leafletjs.com/)** — 지도 렌더링
- **[Leaflet.markercluster](https://github.com/Leaflet/Leaflet.markercluster)** — 마커 클러스터링
- **[Leaflet Routing Machine](http://www.liedman.net/leaflet-routing-machine/)** + OSRM — 경로 안내
- **Google Places API** (초기 데이터 수집 단계에서만) — 실제 장소 데이터(좌표, 카테고리) 수집에 활용
- **GitHub Pages** — 무료 정적 호스팅 (프론트엔드), Supabase가 백엔드 역할

CDN 라이브러리는 unpkg → jsdelivr → cdnjs 순서로 자동 재시도하도록 만들어서, 특정 CDN이 네트워크 환경에 따라 차단되어도 앱이 죽지 않도록 방어적으로 설계했습니다.

### 왜 Leaflet을 선택했나

오픈소스 지도 라이브러리로는 대표적으로 **Leaflet, OpenLayers, MapLibre GL JS** 세 가지가 있습니다.

- **OpenLayers**: GIS 기능이 가장 풍부하고 강력하지만, 그만큼 학습 곡선이 가파르고 무거움
- **MapLibre GL JS**: WebGL 기반 벡터 타일 렌더링으로 부드러운 확대/축소, 3D, 커스텀 스타일링에 강함. Mapbox GL JS의 오픈소스 포크로, 최근 빠르게 채택되고 있음
- **Leaflet**: 가볍고 배우기 쉬우며, 마커·팝업 등 기본 상호작용에 최적화됨

이 프로젝트는 122개 지점에 마커를 찍고 팝업·클러스터링·간단한 경로 안내를 붙이는 정도의 요구사항이라, **Leaflet의 단순함이 오히려 적합**하다고 판단했습니다. 대용량 벡터 타일 렌더링이나 3D 시각화가 필요한 프로젝트라면 MapLibre GL JS를, 복잡한 공간 분석(버퍼, 위상 연산 등)이 필요하다면 OpenLayers를 선택했을 것입니다.

같은 맥락으로, deck.gl·GeoArrow·GeoParquet 같은 도구들은 수백만 건 이상의 대용량 지리공간 데이터를 GPU 기반으로 시각화할 때 강점이 있는 최신 스택입니다. 이번 프로젝트는 122개 지점 규모라 GeoJSON+Leaflet 조합으로 충분하다고 판단해 도입하지 않았지만, 데이터 규모가 커진다면 우선적으로 검토할 기술들입니다. COG(래스터), FlatGeobuf/GeoParquet/PMTiles(벡터), Zarr(다차원 배열) 같은 클라우드 네이티브 지리공간 포맷도 알고 있으나, 122개 규모의 포인트 데이터에는 해당 포맷들의 이점(대용량 스트리밍, 부분 로딩)이 발휘되지 않아 도입하지 않았습니다.

## 데이터베이스 구조

테이블 2개로 구성된 정규화된 구조입니다.

```
places (기준 데이터, 한국어 원본)
  id, name, category, type, lat, lng,
  has_vegan, has_spicy, has_photo_spot, note, price_level, menu

place_translations (번역 데이터, 언어별)
  place_id (FK → places.id), lang, note, menu
  PRIMARY KEY (place_id, lang)
```

`place_id + lang`을 복합 기본키로 잡아서, 장소 하나당 언어별로 한 행씩만 존재하도록 제약했습니다. 새 언어를 추가할 땐 `place_translations`에 행만 추가하면 되고, `places` 테이블 구조는 안 건드려도 됩니다.

## 개발하면서 발견하고 고친 버그

AI(Claude)와 페어 프로그래밍으로 개발하면서, 결과물을 그대로 쓰지 않고 실제로 발생한 문제들을 직접 검증하고 고쳤습니다.

### 1. 마커 클릭/호버가 마커마다 들쭉날쭉하게 작동
- **문제**: 지도 위 색깔 점(마커) 일부는 클릭이 되고, 일부는 아예 반응이 없었음. 겹침 문제인 줄 알았으나, 겹치지 않은 마커도 동일하게 발생.
- **원인 분석**: 마커를 `L.circleMarker`(벡터 도형)로 그리고 있었는데, 마커 클러스터링 라이브러리(`Leaflet.markercluster`)는 원래 `L.marker`(아이콘 기반) 위주로 설계되어 있어서, 벡터 도형을 넣으면 이벤트 처리가 불안정해짐.
- **해결**: 모든 마커를 `L.marker` + CSS로 그린 원(`divIcon`) 방식으로 전면 교체. 클러스터링 라이브러리가 표준적으로 지원하는 방식이라 클릭/호버가 안정적으로 동작하게 됨.

### 2. 리스트 클릭 시 지도 이동은 되는데 팝업이 안 열림
- **문제**: 사이드바 리스트에서 장소를 클릭하면 지도는 이동하는데, 팝업(정보창)이 열리지 않는 경우가 있었음.
- **원인 분석**: 클러스터링 상태에서 마커를 보여주는 `zoomToShowLayer()` 함수가 지도 이동을 전담해야 하는데, 그 직전에 `map.setView()`를 따로 호출해서 두 동작이 충돌 — 콜백(팝업 열기)이 실행되지 않는 경우가 발생.
- **해결**: 지도 이동을 라이브러리에게 완전히 맡기도록 순서 정리. 혹시 모를 예외 상황까지 대비해 1.2초 안전장치(fallback)도 추가.

### 3. GPS 위치를 거부해도 "내 위치 기준"이라고 잘못 표시됨
- **문제**: 사용자가 위치 권한을 거부하면 자동으로 뒤셀도르프 중앙역 좌표로 대체되도록 만들어뒀는데, 이후 화면에는 계속 "내 위치 기준"이라고 표시됨.
- **원인 분석**: "GPS 위치가 있는지"와 "그 위치가 진짜 GPS인지, 대체값인지"를 구분하는 값이 없이 변수 하나로 처리하고 있었음.
- **해결**: `isRealGpsLocation`라는 별도 상태값을 추가해서, 실제 GPS 성공 여부에 따라 라벨(리스트 거리 표시, 코스 출발점 안내, 버튼 문구 등 총 3곳)이 정확하게 갈리도록 수정.

### 4. 즐겨찾기/코스 추가 버튼을 누르면 팝업이 사라짐
- **문제**: 지도 팝업에서 "즐겨찾기"나 "코스에 담기" 버튼을 누르면, 방금 보고 있던 팝업 자체가 사라져버림.
- **원인 분석**: 버튼을 누를 때마다 지도 전체를 처음부터 다시 그리는 함수를 호출하고 있었는데, 이 과정에서 마커 전체가 삭제 후 재생성되면서 열려있던 팝업도 같이 사라짐. (참고로 이 문제는 앱을 버벅이게 만드는 원인이기도 했음)
- **해결**: 전체 재렌더링 대신, 바뀐 버튼/팝업/리스트 항목 하나만 갱신하는 함수(`refreshPopupIfOpen`, `refreshListItem`)로 교체.

## 알고 있는 한계 / 앞으로 개선하고 싶은 점

실무 프로덕션 코드였다면 아래처럼 진행했을 것입니다 (지금은 단일 파일 프로젝트 특성상 의도적으로 타협한 부분들입니다):

- **상태 관리**: 지금은 모듈 스코프의 전역 변수로 상태를 관리합니다. React 등으로 옮긴다면 컴포넌트 상태/컨텍스트로 정리하고 싶습니다.
- **모듈 분리**: `script.js` 하나에 모든 로직이 들어있습니다. 기능별로 파일을 나누고 번들러(Vite 등)를 도입하면 유지보수가 쉬워질 것입니다.
- **접근성**: 주요 버튼에 `aria-label`을 추가했지만, 스크린 리더 전체 흐름 테스트는 아직 못 했습니다.
- **데이터 정확도**: 122곳의 좌표는 Google Places API 기반이라 정확하지만, 비건/매운맛 여부·가격대·대표 메뉴 일부는 리뷰 기반 추정치입니다. 직접 방문 검증한 곳부터 순차적으로 업데이트할 계획입니다.
